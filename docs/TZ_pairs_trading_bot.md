# ТЗ: Система парного стат-арбитража (BTC/ETH перпетуалы) + веб-панель

> **Назначение документа.** Это техническое задание для разработки (в т.ч. с помощью Claude/Claude Code). Оно сознательно отходит от исходного промта в местах, где тот содержит дефекты или нереализуемые на MEXC требования. Биржа вынесена в абстракцию через CCXT, чтобы исполнять на площадке с открытым фьючерсным API.

---

## 0. Контекст и критические ограничения

- **MEXC futures API закрыт для розницы.** Размещение ордеров на контрактах через API доступно фактически только участникам Market Maker программы. Розничный аккаунт получает ошибку `code 1002 "Contract not allow place order!"`. Поэтому:
  - **Чтение** рыночных данных с MEXC — допустимо.
  - **Исполнение** — на бирже с открытым perp-API: **Bybit (рекомендуется), OKX, Gate.io, Bitget, Binance Futures**.
  - Биржа задаётся конфигом; код не должен быть жёстко привязан к MEXC.
- Если котировки берутся с одной биржи, а исполнение на другой — возникает **базисный риск**. По умолчанию: **считать и исполнять на одной бирже** (Bybit).
- **Это не финансовая рекомендация.** Система — инструмент; ответственность за параметры и капитал на операторе.

---

## 1. Цель

Автоматизированная торговля парной mean-reversion стратегией на двух перпетуалах (BTCUSDT, ETHUSDT, USDT-маржа) с веб-панелью мониторинга и управления. Браузер логику **не** исполняет — только наблюдает и шлёт команды.

---

## 2. Торговая стратегия (исправленная)

### 2.1 Сигнальный ряд
- Базовый таймфрейм: **5m**, расчёт **только по закрытым свечам** (без repaint).
- Спред: `spread = ln(P_BTC) − β · ln(P_ETH)`, где `β` — динамический хедж-коэффициент (rolling OLS или Kalman, окно настраиваемо; дефолт — OLS на 240 барах). Допускается упрощённый режим `ratio = P_BTC / P_ETH` как опция конфига.
- Индикатор: **Bollinger Bands**, период **240** (SMA), множитель **2σ**, поверх `spread`. Эквивалентно — **z-score** спреда; пороги задаются в z (дефолт: вход |z| ≥ 2, выход |z| ≤ 0.2).

### 2.2 Фильтр волатильности (анти-флэт)
- Не входить, если ширина канала `(upper − middle) / middle < THRESHOLD` (дефолт 2%, настраиваемо). Слишком узкий канал = нет статистического эдва.

### 2.3 Вход
- Условие: `spread` касается/пробивает **нижнюю полосу** (z ≤ −2) **и** пройден фильтр волатильности **и** нет открытой позиции по этому сигналу (idempotency).
- Действие: **лонг BTCUSDT-perp + шорт ETHUSDT-perp**, **доллар-нейтрально** (равный нотационал в USDT на каждую ногу, с учётом `β`).

### 2.4 Выход
- Возврат `spread` к средней линии (z ≥ −0.2 / касание middle) → **закрыть обе ноги**.

### 2.5 Риск-менеджмент
- Сайзинг: суммарный нотационал позиции ≤ **2%** доступного USDT во фьючерсном кошельке (параметр; явно указать — это нотационал, не риск).
- Стоп: по **z-score** (дефолт |z| ≥ 3.5), а **не** жёсткие 0.5% от цены (0.5% для BB(240)/5m выбивает почти всегда — этот пункт промта отклонён). Дополнительно — стоп по времени в позиции (дефолт 48 ч).
- Плечо: **1x–2x**, isolated margin.
- Один открытый набор позиций на сигнал; повторный вход запрещён до закрытия.
- Kill-switch: ручная и авто-остановка (дневной лимит убытка).

---

## 3. Исполнение (OMS)

- **Лимитные ордера** с контролем проскальзывания ≤ 0.2%.
- Таймаут исполнения **5 с**: не залилось — отмена и переустановка с корректировкой цены (макс. N попыток, затем — рынок или отмена цикла).
- **Защита от leg-in риска:** если за таймаут залилась только одна нога — добить вторую рынком в пределах допуска проскальзывания **или** откатить первую. Состояние «полузаполнено» недопустимо как стабильное.
- Idempotent client order id; дедупликация при реконнекте.
- Учёт комиссий (maker/taker) и **funding** обеих ног в P&L.

---

## 4. Логирование (требование сохранено и расширено)

На каждое событие писать структурный лог (JSON) + в БД:
- дата/время (UTC, ISO-8601);
- цены BTC, ETH, значение `spread`, `β`, `z`;
- полосы BB: upper / middle / lower;
- ширина канала, %;
- действие: вход/выход, направление по каждой ноге, объём (контракты + нотационал USDT), цена исполнения, комиссия, slippage;
- баланс до/после, текущий unrealized PnL.

## 5. Архитектура

Event-driven async core, 4 слоя, браузер вне торгового контура.

```
Биржа(WS, ccxt.pro) → Bot worker(asyncio) ⇄ Redis ⇄ FastAPI(REST+WS) → Next.js dashboard
                              │ write
                       PostgreSQL + TimescaleDB
```

- **Data layer:** WS-стримы (ccxt.pro), сборка 5m баров, расчёт спреда/β/BB.
- **Strategy engine:** FSM `FLAT → ENTERING → IN_POSITION → EXITING`.
- **Execution/OMS:** см. §3.
- **Risk layer:** сайзинг, стопы, лимиты, kill-switch.
- **Persistence:** PostgreSQL+TimescaleDB (бары/сделки/логи), Redis (live-state + шина).
- **API:** FastAPI (REST для команд, WS для real-time).
- **Frontend:** Next.js + TS + TanStack Query + Tailwind + shadcn/ui; графики — TradingView Lightweight Charts (свечи спреда + оверлей BB); метрики — Recharts.

## 7. Функциональные требования веб-панели

- Дашборд: live-спред + BB, текущее состояние FSM, открытые позиции (обе ноги), PnL (realized/unrealized), funding.
- Журнал сделок и событий (фильтры по дате/действию).
- Управление: start / stop / **flatten (закрыть всё рынком)**, редактирование параметров стратегии (период BB, σ, пороги z, фильтр ширины, сайзинг, плечо, таймауты).
- Индикаторы здоровья: WS-коннект, лаг данных, ошибки API, баланс.
- Аутентификация (минимум — единый оператор, JWT), read-only режим.

---

## 8. Нефункциональные требования

- Реакция на сигнал по закрытию свечи: < 1 с.
- Переживать реконнект WS и рестарт процесса без потери состояния (восстановление из БД/Redis).
- Полный аудит-трейл всех ордеров.
- **Обязателен бэктест-режим** на исторических данных + **paper-trading** до реальных денег.
- Конфиг — единый валидируемый pydantic-объект, без хардкода ключей/символов/биржи.

---

## 9. Этапы

1. **MVP данных:** стрим свечей, расчёт спреда/β/BB/z, отрисовка на дашборде (без торговли).
2. **Бэктест** стратегии на истории, отчёт по метрикам (Sharpe, max DD, win-rate, кол-во сделок, влияние комиссий/funding).
3. **Paper-trading** через тестнет/эмуляцию OMS.
4. **Live** на малом капитале с kill-switch.
5. Панель управления параметрами + алерты (Telegram).
