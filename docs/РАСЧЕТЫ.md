# Логика расчётов — для проверки

Описание всех формул стратегии с привязкой к коду. Соответствует ТЕКУЩЕМУ коду
(rolling-β по доходностям — рабочее состояние). Параметры по умолчанию из `config.py`.
Все расчёты — по **закрытым** свечам (no repaint).

Поток данных: `data → indicators → SignalEngine → Engine(approve/reject) → VirtualExchange → summary`

---

## 0. Входные данные

Две цены закрытия, выровненные по времени:
- `price_a` — нога A (по умолчанию XRP/USDT перп)
- `price_b` — нога B (по умолчанию ADA/USDT перп)

Источник: CCXT с MEXC (только чтение) либо синтетический генератор.

---

## 1. Хедж-β (коэффициент пары)
Файл: `indicators.py` → `rolling_beta()`

Считается по **лог-доходностям** на **скользящем** окне `beta_window` = 240:

```
ra = Δ ln(price_a)              # лог-доходности ноги A
rb = Δ ln(price_b)              # лог-доходности ноги B
β  = cov(ra, rb) / var(rb)      # rolling, окно 240
```

- OLS-наклон регрессии доходностей A на доходности B (классический hedge ratio).
- По доходностям, а не по уровням лог-цен: на уровнях в боковике `var → 0`, β скачет
  (std ~0.5) и уходит в минус. По доходностям β стабильна (std ~0.02, без отрицательных).
- Первые 240 баров β = NaN (прогрев).

---

## 2. Спред
Файл: `indicators.py` → `build_indicators()`

Режим `log` (по умолчанию):
```
spread = ln(price_a) − β · ln(price_b)
```
Режим `ratio` (β = 1):
```
spread = price_a / price_b
```

---

## 3. Полосы Боллинджера поверх спреда
Окно `bb_period` = 240, `bb_k` = 2.0:
```
mid   = SMA(spread, 240)            # скользящая средняя спреда
std   = STD(spread, 240, ddof=0)    # скользящее СКО (популяционное, ddof=0)
upper = mid + bb_k · std
lower = mid − bb_k · std
```

---

## 4. z-score
```
z = (spread − mid) / std
```
z — отклонение спреда от средней в единицах σ. z = ±2 → у полосы; z = 0 → на средней.

---

## 5. Ширина канала (анти-флэт фильтр)
```
width_pct = (bb_k · std) / |mid| · 100%
```
Вход запрещён, если `width_pct < min_width_pct` (2%).

---

## 6. Сигналы входа/выхода
Файл: `strategy.py` → `SignalEngine.evaluate()`

### Вход (рекомендация; авто-вход при auto_approve)
По умолчанию entry_z=2, stop_z=3.5, min_width_pct=2:
```
width_pct < min_width_pct                  → НЕТ (флэт)
−stop_z < z ≤ −entry_z                      → ВХОД, LONG_SPREAD  (лонг A / шорт B)
 entry_z ≤ z <  stop_z (и allow_short)      → ВХОД, SHORT_SPREAD (шорт A / лонг B)
иначе                                       → НЕТ
```
- **LONG_SPREAD** — спред внизу (у нижней полосы), ставка на рост спреда к средней.
- **SHORT_SPREAD** — спред вверху, ставка на падение спреда к средней.
- Фильтр `|z| < stop_z`: не входим, если спред уже за стоп-уровнем.

### Выход (авто) — по тому же динамическому z
```
ВОЗВРАТ К СРЕДНЕЙ (reason="exit"):
  LONG_SPREAD:  z ≥ −exit_z          (exit_z = 0.2)
  SHORT_SPREAD: z ≤  exit_z
СТОП по z (reason="stop"):
  LONG_SPREAD:  z ≤ −stop_z          (stop_z = 3.5)
  SHORT_SPREAD: z ≥  stop_z
ТАЙМ-СТОП (reason="stop"):
  bars_held ≥ max_bars_in_trade      (576 ≈ 48ч на 5m)
```
Выход меряется тем же `z`, что и вход (раздел 4) — в той же rolling-β.

## 7. Сайзинг позиции (доллар-нейтральный)
Файл: `virtual_exchange.py` → `open_pair()`

```
notional   = balance · risk_pct / 100        # risk_pct = 2%
notional_a = notional / (1 + β)              # нога A
notional_b = notional − notional_a           # нога B  →  notional_b / notional_a = β
```
Исполнение с проскальзыванием (slippage = 0.02%):
```
fill = ref · (1 + slippage)   при покупке (long)
fill = ref · (1 − slippage)   при продаже (short)
qty  = notional_ноги / fill                  # количество контрактов
```
Комиссия входа: `(notional_a + notional_b) · taker_fee`, taker_fee = 0.06% за ногу.

`spread0 = ln(price_a) − β · ln(price_b)` — спред входа (для отметок на графике).

## 8. P&L (прибыль/убыток)
Файл: `virtual_exchange.py` → `close_pair()`, `unrealized()`

### По каждой ноге
```
знак = +1 если long, −1 если short
pnl_нога = qty · (цена_выхода − цена_входа) · знак
```
Long зарабатывает на росте цены, short — на падении. Цены выхода тоже со слиппеджем.

### Валовый и чистый
```
gross    = pnl_a + pnl_b
exit_fee = (qty_a·fa + qty_b·fb) · taker_fee
net      = gross − exit_fee          # комиссия входа уже списана при открытии
fees     = entry_fee + exit_fee      # суммарные комиссии (вход + выход)
```

### Сводные метрики (`engine.py` → summary)
```
net_pnl    = Σ net по закрытым сделкам
win_rate   = доля сделок с net > 0
equity     = balance + нереализованный P&L открытой позиции
return_pct = (equity − start_balance) / start_balance · 100%
```

## 9. Логика «вход у границы → выход у средней»

Геометрия (LONG_SPREAD):
- вход: z = −2 → спред на 2σ **ниже** средней.
- выход: z ≈ 0 → спред **на** средней → вырос на ~2σ.
- спред вырос = `ln(A) − β·ln(B)` вырос → лонг A / шорт B в плюсе.

**Важная оговорка (текущее состояние, rolling-β):** β пересчитывается каждый бар.
Знак изменения спреда зависит от β, а P&L по ногам считается в β входа. При заметном
дрейфе β между входом и выходом возможны случаи, когда z вернулся к средней (выход),
но валовый P&L отрицателен — потому что скользящая средняя «уехала» вслед за спредом.
На синтетике таких ~5 из 19. Это известное свойство mean-reversion на скользящей
средней, не баг расчёта P&L (P&L по ногам всегда корректен).

Остаточный `net < 0` также возможен из-за комиссий+слиппеджа (round-trip ≈
2·(0.06% + 0.02%) = 0.16% нотационала), когда ход спреда меньше издержек.

## ⚠️ Что проверить особенно внимательно

1. **β по доходностям, спред по уровням** (`ln A − β·ln B`). β из пространства
   приращений, спред — из уровней. Методологическое смешение; работает эмпирически,
   но строго коинтеграционная β считалась бы по уровням.

2. **Выход по z, а не по фиксированному уровню.** Выход срабатывает при `|z| ≤ exit_z`
   относительно СКОЛЬЗЯЩЕЙ средней. Если средняя «уехала», z вернётся к нулю без
   реального возврата спреда → возможен мелкий минус (см. раздел 9).

3. **rolling-β дрейфует.** P&L считается в β входа; за время сделки β меняется.
   Пробовали фиксировать β / expanding-β — оба варианта откатили как неверно
   работающие. Текущее (rolling) — подтверждённое рабочее состояние.

4. **Метрики синтетики ≠ реальная доходность** (записано в CLAUDE.md). Для оценки
   прибыльности нужен бэктест на длинной реальной истории.
