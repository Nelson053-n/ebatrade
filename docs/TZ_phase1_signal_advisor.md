# ТЗ — Phase 1: Аналитик-советник по парному спреду + виртуальная биржа

> Документ для Claude Design / разработки. Соответствует готовому скелету `pairsignal/`.
> Scope phase 1: **читаем данные, считаем, советуем, оператор решает, исполняем виртуально.**
> С реальной биржей по торгам **не работаем.**

## 1. Цель
Система указанных оператором пар инструментов: собирает котировки (только чтение),
считает статистику спреда, выдаёт **рекомендации** на вход, ждёт решения оператора,
ведёт **виртуальные** сделки и закрывает их по заданным условиям. Никаких реальных ордеров.

## 2. Границы
- ✅ Чтение публичных OHLCV через CCXT (ключи не нужны; для MEXC чтение разрешено).
- ✅ Расчёт индикаторов, генерация сигналов, виртуальное исполнение, учёт P&L.
- ✅ Human-in-the-loop: вход — только после подтверждения оператора.
- ❌ Реальное размещение ордеров (отдельная поздняя фаза, после устойчивого paper-результата).

## 3. Вход оператора
- Список пар: `symbol_a`, `symbol_b` (CCXT-формат перпетуала, напр. `BTC/USDT:USDT`).
- Биржа данных, таймфрейм, режим спреда (`log`/`ratio`), пороги (`entry_z`, `exit_z`,
  `stop_z`, `min_width_pct`, `max_bars_in_trade`), параметры paper (баланс, `risk_pct`,
  комиссия, проскальзывание). Всё — в `config.py` (pydantic), без хардкода.

## 4. Аналитика и сигналы
- Спред: `ln(A) − β·ln(B)` (динамическая β, rolling-OLS, окно `beta_window`) или `A/B`.
- Bollinger(`bb_period`, `bb_k`) поверх спреда → `mid/upper/lower/std`, **z-score**, ширина %.
- Все расчёты по **закрытым** свечам (no repaint).
- **Рекомендация на вход** (требует подтверждения): `−stop_z < z ≤ −entry_z` → лонг A / шорт B;
  симметрично сверху → шорт A / лонг B. Гейт: ширина канала ≥ `min_width_pct` (анти-флэт).
- **Выход (авто):** возврат к средней `|z| ≤ exit_z`.
- **Стоп (авто):** `|z| ≥ stop_z` или превышен `max_bars_in_trade`.

## 5. Виртуальная биржа
- Доллар-нейтральный сайзинг: нотационал = `risk_pct`% баланса, распределение ног по β.
- Симуляция филлов с проскальзыванием, комиссия за вход и выход.
- Учёт реализованного и нереализованного P&L, журнал закрытых сделок.

## 6. Сводка/метрики
Кол-во сделок, win-rate, net P&L, суммарные комиссии, баланс, equity, доходность %.
(Дальше — Sharpe, max DD, кривая капитала.)

## 7. Логирование (на каждое событие)
Время UTC, цены A/B, спред, β, z, полосы BB, ширина %, действие (вход/выход/стоп),
направление по ногам, объём (контракты + нотационал), цена исполнения, комиссия, P&L.

## 8. Backend (под панель)
FastAPI: `GET /state` (рекомендация, позиция, сводка), `POST /approve|/reject`
(решение оператора), `POST /live/start|/stop` (фоновый опрос котировок),
`POST /replay/synthetic` (демо). Real-time на фронт — добавить WebSocket.

## 9. Архитектура (соответствие скелету)
```
CCXT(read) / synthetic → indicators → SignalEngine(FSM) → Engine(approve/reject)
                                                              → VirtualExchange(paper) → summary/trades
                                              FastAPI(api.py) ⇄ оператор/панель
```
Слои изолированы: данные / индикаторы / сигнал / исполнение / оркестрация / API.

## 10. Стек
Python 3.12, ccxt, pandas/numpy, pydantic v2, FastAPI/uvicorn.
Фронт (следующий шаг): Next.js + TS + Lightweight Charts поверх `/state` + WS.

## 11. Критерии готовности phase 1
- Запуск `python -m pairsignal.main --synthetic` даёт корректные сигналы и сводку.
- `--live` тянет реальные свечи и строит те же индикаторы.
- Вход исполняется только после `approve`; выход/стоп — автоматически по правилам.
- Полный журнал сделок и метрики доступны через CLI и `GET /state`.

## 12. Следующие фазы (вне scope)
Бэктест-отчёт и оптимизация параметров → Telegram-алерты → веб-панель (Next.js+WS)
→ только потом реальное исполнение на бирже с открытым фьючерсным API.
